I. Introduction: Project Plan¶

For this project, I will be using the aviation accident dataset from PlaneCrashInfo.com and the OpenFlights Airports Database to analyze patterns in the frequency, correlations, and outcomes of plane crashes. Focusing specifically on the year 1985, I aim to examine aviation accident trends and outcomes during that period. My primary objective is to explore how accident frequency and severity correlate with distance from the nearest airport as well as different aircraft manufacturers and specific time factors such as the month and day of occurrence.

Consideration of the Dataset¶

PlaneCrashInfo.com provides a comprehensive aviation accident database covering the years 1920 to 2024, encompassing all types of accidents—civil, commercial, cargo, test flights, military transports, commercial and military helicopters, and airships. Each yearly dataset includes information on the date, location, operator, aircraft manufacturer, aircraft type, registration, and fatalities. For this analysis, the 1985 dataset offers sufficient detail for conducting basic statistical evaluations.

The OpenFlights Airports Database is a comprehensive dataset containing information about airports worldwide. It includes details such as the airport's name, location (city and country), geographical coordinates (latitude and longitude), IATA and ICAO codes, altitude, time zone, and airport type. This dataset provides a valuable resource for analyzing global air traffic patterns, geographic distribution of airports, and potential relationships between airport locations and aviation accidents. For this analysis, the OpenFlights Airports Database is used to identify the nearest airports to aviation accidents, helping to explore proximity-based patterns and insights in the accident data.

import requests
from bs4 import BeautifulSoup
import pandas as pd

# This is the function for scraping and parsing the aviation accident data from planecrashinfo.com
def scrape_planecrash_data(url):
    # Sending GET request to fetch the webpage content
    response = requests.get(url)

    # Using BeautifulSoup to parse the HTML content of the page
    soup = BeautifulSoup(response.content, 'html.parser')

    # Finding the first <table> tag on the page that contains the relevant data
    table = soup.find('table')

    # Reading the HTML table into a pandas DataFrame
    df = pd.read_html(str(table))[0]

    # Cleaning up the DataFrame by setting the first row as the column names
    df.columns = ['Attribute', 'Value']  # Assigning proper column names
    df = df.dropna()  # Dropping any rows with missing values

    # Pivoting the data so that each attribute becomes a column
    df_clean = df.set_index('Attribute').T  # Transposing the DataFrame to turn attributes into columns
    return df_clean

# Listing the years (pages) to scrape
ranges = range(1, 56)  # Pages 1 to 56

# Creating a list to store DataFrames for each page
dfs = []

# Looping through each page to scrape the data
for r in ranges:
    # Constructing the URL for each page
    url = f'http://planecrashinfo.com/1985/1985-{r}.htm'
    try:
        # Scraping the data and appending the resulting DataFrame
        df = scrape_planecrash_data(url)
        dfs.append(df)
    except Exception as e:
        # Printing the URL and error message if an error occurs
        print(f"Failed to scrape {url}: {e}")

# Concatenating all DataFrames into one DataFrame
df_1985 = pd.concat(dfs, ignore_index=True)

# Saving the cleaned data to a CSV file for further analysis
df_1985.to_csv('planecrash_1985.csv', index=False)

ETL: Preparing the Location Column for Geographic Analysis¶

To prepare the Location column for extracting geographical coordinates, the data must first be standardized. This involves cleaning up inconsistencies in location names, such as removing unnecessary qualifiers (e.g., "Near," "Off," or distance estimates) and ensuring uniformity in city and country names. For instance, locations like "Near La Paz, Bolivia" should be transformed to just "La Paz, Bolivia," and ambiguous descriptions like "300 nm NW of San Francisco, California" should be simplified to "San Francisco, California." Once these updates are made, the Location column will contain standardized place names that can be reliably used to extract latitude and longitude for further analysis. This step is critical for ensuring that the geographic data can be accurately mapped and analyzed in subsequent stages.

ETL: Extracting Latitude and Longitude from Location Data¶

To facilitate geographic analysis, the latitude and longitude coordinates for each location in the Location: column are extracted using the Geopy library. By geocoding the cleaned location names, we can map the places to their geographical coordinates (latitude and longitude). This process is crucial for spatial analysis and visualizing accident locations on a map.

# Importing the necessary geocoder module from geopy
from geopy.geocoders import Nominatim

# Initializing the geocoder with a custom user agent
geolocator = Nominatim(user_agent="plane_crash_geocoder")

# Function to get latitude and longitude for a given location
def get_lat_lon(location):
    try:
        # Geocoding the location (city, country) to get geographical coordinates
        location_info = geolocator.geocode(location)
        if location_info:
            # Returning latitude and longitude if geocoding is successful
            return location_info.latitude, location_info.longitude
        else:
            # Returning None if location could not be geocoded
            return None, None
    except Exception as e:
        # Handling any errors during geocoding and printing the error message
        print(f"Error geocoding {location}: {e}")
        return None, None

ETL: Finding the Nearest Airport to Each Crash Location¶

To associate each aviation accident with the nearest airport, the latitude and longitude of the crash locations are compared with the coordinates of airports from the OpenFlights dataset. The geodesic method from the Geopy library is used to calculate the distance between the crash site and each airport, identifying the closest airport for each accident. This step adds valuable geographic context to the crash data.

In [5]:
from geopy.distance import geodesic

# Function to find the nearest airport based on latitude and longitude of the crash
def find_nearest_airport(crash_lat, crash_lon, airports_df):
    # Initializing a variable to store the minimum distance (start with infinity)
    min_distance = float('inf')
    nearest_airport = None

    # Looping through each airport in the airports DataFrame
    for _, row in airports_df.iterrows():
        airport_lat = row['Latitude']
        airport_lon = row['Longitude']

        # Calculating the geodesic distance between the crash location and the airport
        distance = geodesic((crash_lat, crash_lon), (airport_lat, airport_lon)).kilometers

        # If this airport is closer, update the minimum distance and nearest airport
        if distance < min_distance:
            min_distance = distance
            nearest_airport = row  # Store the entire row for airport details

    return nearest_airport

# Initializing new columns in the DataFrame to store the nearest airport details
df_1985['Nearest_Airport'] = None
df_1985['Nearest_Airport_Latitude'] = None
df_1985['Nearest_Airport_Longitude'] = None

# Loop through each crash in the dataset to find the nearest airport
for idx, row in df_1985.iterrows():
    crash_lat = row['Latitude']
    crash_lon = row['Longitude']

    # Find the nearest airport for each crash based on latitude and longitude
    nearest_airport = find_nearest_airport(crash_lat, crash_lon, airports_df)

III. Exploratory Data Analysis (EDA)¶

Frequency of Crashes by Month¶

To explore the frequency of crashes by month, the Month column is extracted from the Date column. The number of crashes for each month is then counted, and the month with the highest number of crashes is identified, along with the month with the fewest crashes. This analysis reveals the months with the most and least crashes, providing valuable insights into the distribution of accidents over the course of the year.

In [13]:
# Extract the month from the 'Date' column
df_1985['Month'] = df_1985['Date'].dt.month

# Group by month and count the number of crashes
month_counts = df_1985['Month'].value_counts()

# Find the month with the most crashes (max)
max_month = month_counts.idxmax()  # Month with the highest number of crashes
max_month_count = month_counts[max_month]  # Number of crashes in the most frequent month

# Find the month with the fewest crashes (min)
min_month = month_counts.idxmin()  # Month with the fewest crashes
min_month_count = month_counts[min_month]  # Number of crashes in the least frequent month

# Display the results
print(f"Month with the most crashes: {max_month}st with {max_month_count} crashes")
print(f"Month with the fewest crashes: {min_month}rd with {min_month_count} crashes")

Calculating and Analyzing the Distance to Nearest Airport¶

In order to understand the proximity of aviation crashes to their nearest airports, the distance between each crash location and its nearest airport is calculated using the geodesic distance formula. This calculation uses the latitude and longitude of both the crash and the nearest airport to determine the distance in kilometers. Once the distance is calculated, it is added to the DataFrame as a new column called Distance to Nearest Airport.

By applying this function across the dataset, the distances are computed for each crash and summarized using descriptive statistics. These summary statistics (such as mean, median, and standard deviation) provide insights into how far crashes typically occur from airports, helping to identify any significant patterns or outliers in proximity.

In [17]:
# Function to calculate distance between crash location and nearest airport
def calculate_distance(crash_lat, crash_lon, airport_lat, airport_lon):
    return geodesic((crash_lat, crash_lon), (airport_lat, airport_lon)).kilometers

# Apply the function to each row to create a new column with the distances
df_1985['Distance to Nearest Airport'] = df_1985.apply(
    lambda row: calculate_distance(row['Latitude'], row['Longitude'], row['Nearest Airport Latitude'], row['Nearest Airport Longitude']), axis=1)

VI. Data Challenges¶

Issues with the Datasets: PlaneCrashInfo.com and OpenFlights Data¶

The aviation accident dataset for 1985 provides valuable insights, but several limitations need to be considered regarding the accuracy of the source and the completeness of the data. The website PlaneCrashInfo.com, which serves as the primary source for the dataset, includes a disclaimer about the validity of the information provided. Despite efforts to verify the data from multiple sources, the website does not guarantee its accuracy, and the dataset does not claim to represent the conclusions of national aviation authorities.

Moreover, when integrating the OpenFlights Airports Database to assess the proximity of airports to crash locations, challenges can arise. While OpenFlights provides useful information on airport locations, inconsistencies in airport names or missing data for smaller regional airports can cause issues when merging datasets. Any discrepancies in the airport data may lead to inaccuracies in spatial analyses, particularly in terms of crash proximity to airports. Validation of the OpenFlights data is crucial to ensure consistency and reliability when combining it with the crash data.

Issues with the Dataset: Month, Day, and Operator Analysis¶

The dataset's focus on only one year, 1985, presents challenges when trying to assess trends based on the month or day of the crash. Without controlling for confounding variables, such as weather, airline safety records, or flight duration, conclusions drawn from month and day analyses may not be definitive. For example, some months or days may have more crashes simply due to factors like higher air traffic during peak seasons or specific operational challenges. Additionally, analyzing operators, such as Japan Air Lines, in terms of fatalities is problematic because the dataset does not account for the total number of flights conducted by each operator. The high number of fatalities associated with certain operators may simply reflect their larger share of the aviation market, rather than indicating greater risk. To make these analyses more meaningful, data from multiple years should be incorporated to control for seasonal factors, and a more comprehensive analysis of operator safety would require factoring in the total flight volume or flight hours for each operator.

Issues with the Dataset: Distance and Correlation Analysis¶

When calculating the distance to the nearest airport and the correlation between crash locations and nearest airports, several issues arise. The geospatial analysis assumes that the nearest airport is the most relevant one for the crash location, but this may not always be the case. For example, crashes may occur in remote areas with few nearby airports, and the closest airport may not have been operational at the time or accessible for rescue operations. Additionally, the correlation analysis between crash locations and airport locations assumes a linear relationship, but crashes could be randomly distributed or influenced by other unmeasured variables, such as air traffic volume or the regional distribution of airports. The assumptions underlying these analyses should be tested, and caution should be taken in interpreting the results, as the proximity to an airport may not fully explain the likelihood of a crash occurring at a given location.

Issues with the Dataset: Missing Data and Causal Analysis¶

A significant challenge in the dataset is the presence of missing data, which limits the accuracy and depth of any analysis. Incomplete rows and missing crash details, such as the causes of accidents, prevent comprehensive causal analysis. Although each crash has a summary, there is insufficient detail on the contributing factors, making it difficult to fully assess the causes of aviation accidents. As a result, any relationships derived from this data, such as the correlation between crash locations and nearest airports, may not provide a complete or accurate picture of the factors at play. To fully understand the causes of crashes, additional data sources or more detailed information on accident causes would be necessary.

            'classic'. You can specify the flavor of the format used.

            > jupyter nbconvert --to html --template lab mynotebook.ipynb

            You can also pipe the output to stdout, rather than a file

            > jupyter nbconvert mynotebook.ipynb --stdout

            PDF is generated via latex

            > jupyter nbconvert mynotebook.ipynb --to pdf

            You can get (and serve) a Reveal.js-powered slideshow

            > jupyter nbconvert myslides.ipynb --to slides --post serve

            Multiple notebooks can be given at the command line in a couple of
            different ways:

            > jupyter nbconvert notebook*.ipynb
            > jupyter nbconvert notebook1.ipynb notebook2.ipynb

            or you can specify the notebooks list in a config file, containing::

                c.NbConvertApp.notebooks = ["my_notebook.ipynb"]

            > jupyter nbconvert --config mycfg.py

To see all available configurables, use `--help-all`.
